# Битрикс24: локальное серверное приложение с UI — раздел «Сценарии»

Интеграция панели **Sales AI** (раздел **Сценарии**: QA-аналитика, ИИ-тренер, ИИ-лидогенератор) в портал [Битрикс24](https://www.bitrix24.ru/) как **локальное приложение** с пользовательским интерфейсом (iframe).

Ваш портал: **https://b24-oc3ne4.bitrix24.ru**

---

## Что уже есть в проекте

| Компонент | Назначение |
|-----------|------------|
| **SPA** (`frontend/spa`) | Интерфейс; маршрут **`/bitrix`** редиректит в **`/scenarios/qa-analytics`** с сохранением query-параметров Битрикса (`DOMAIN`, `APP_SID`, `AUTH_ID` …). Можно в настройках приложения указать вместо этого **прямой** URL на **`/scenarios/qa-analytics`**. |
| **`useBitrixAuth`** | Читает параметры из URL и передаёт их в API через заголовки `X-Bitrix-Domain`, `X-Bitrix-App-Sid`, `X-Bitrix-Auth-Id` (см. `frontend/spa/src/api/client.js`). |
| **Backend** `POST /api/bitrix/install` | Приём установки приложения, сохранение OAuth-токенов портала. |
| **Backend** `POST /api/bitrix/events` | События Битрикс24 (проверка `BITRIX24_APPLICATION_TOKEN`). |
| **Nginx** (`frontend/nginx.conf`) | `Content-Security-Policy: frame-ancestors … *.bitrix24.ru …` — без этого iframe из облака Битрикс24 **не откроет** вашу страницу. |

Переменные окружения: см. `.env.example` (`BITRIX24_OAUTH_*`, `BITRIX24_APPLICATION_TOKEN`).

---

## Требования

1. **Публичный HTTPS** для фронтенда и API (Битрикс24 обращается к вашему серверу из облака).
2. Один «внешний» базовый URL SPA, например: `https://app.ваш-домен.ru`.
3. URL API с префиксом `/api/`, например: `https://app.ваш-домен.ru/api/` или отдельный поддомен.

Ниже в примерах подставьте свои хосты вместо `https://app.example.com`.

## Шаг 1. Создать локальное приложение в Битрикс24

1. Войдите на портал **https://b24-oc3ne4.bitrix24.ru** под администратором.
2. Откройте раздел для разработчиков:
   - **Приложения** → **Разработчикам** → **Другое** → **Локальное приложение**  
   - или **Настройки** → **Настройки продукта** → **Разработчикам** → **Локальное приложение**  
   (названия пунктов могут слегка отличаться в зависимости от версии портала).
3. Нажмите **Добавить приложение** / **Создать**.

Заполните поля (формулировки в интерфейсе могут быть на русском):

| Поле | Что указать |
|------|-------------|
| **Название** | Например: `Sales AI — Сценарии` |
| **Код (идентификатор)** | Латиница без пробелов, например: `sales_ai_scenarios` |
| **Тип** | **Серверное** с **пользовательским интерфейсом** (локальное приложение с UI / встраивание в iframe). |

## Шаг 2. URL интерфейса (главное для раздела «Сценарии»)

Укажите адрес, который откроется **внутри iframe** в Битрикс24:

**Вариант A (рекомендуется):** короткая точка входа с редиректом в сценарии:

```text
https://app.example.com/bitrix
```

**Вариант B:** сразу страница сценариев:

```text
https://app.example.com/scenarios/qa-analytics
```

Битрикс24 к этому URL **сам добавит** параметры (`DOMAIN`, `PROTOCOL`, `LANG`, `AUTH_ID`, `REFRESH_ID`, `APP_SID` и др.). Их обрабатывает `useBitrixAuth`; редирект `/bitrix` → `/scenarios/qa-analytics` **сохраняет** query-строку.

---

## Шаг 3. URL установки (handler / install)

Укажите обработчик первичной установки (должен быть доступен метод **POST** из интернета):

```text
https://app.example.com/api/bitrix/install
```

Если API на другом хосте:

```text
https://api.example.com/api/bitrix/install
```

После установки бэкенд сохраняет OAuth-данные портала в таблице `bitrix_portals`.

---

## Шаг 4. OAuth: ключи в `.env` бэкенда

В карточке локального приложения Битрикс24 скопируйте:

- **ключ приложения** / **Client ID** → `BITRIX24_OAUTH_CLIENT_ID`
- **секрет приложения** / **Client secret** → `BITRIX24_OAUTH_CLIENT_SECRET`

При необходимости задайте **ключ проверки** для исходящих запросов/событий → `BITRIX24_APPLICATION_TOKEN` (используется в `POST /api/bitrix/events`).

Перезапустите сервис API после изменения `.env`.

---

## Шаг 5. Права приложения

В настройках приложения отметьте права, которые реально нужны вашей логике (лиды, CRM, пользователи и т.д.). Минимальный набор зависит от сценариев на бэкенде; для только открытия UI в iframe достаточно базовых прав, для REST-вызовов — расширить по [документации REST Битрикс24](https://apidocs.bitrix24.ru/).

---

## Шаг 6. Nginx / CSP

Убедитесь, что для отдачи SPA заданы:

- `Content-Security-Policy` с `frame-ancestors`, включающим ваш портал и зону `*.bitrix24.ru` (пример в `frontend/nginx.conf`).
- Не выставляйте `X-Frame-Options: DENY` для страниц SPA (или снимите его для `location /`).

---

## Шаг 7. Вход пользователя в панель

Сейчас защищённые маршруты требуют **JWT** портала (логин/пароль в вашей форме). При открытии приложения из Битрикс24 пользователь попадёт на **страницу входа**, после успешного входа откроется **`/scenarios/qa-analytics`** с сохранением параметров iframe (в т.ч. если редирект шёл через `/bitrix`).

Дальнейшая доработка (по желанию): связать `AUTH_ID` Битрикс24 с пользователями портала или внедрить отдельный поток доверия к сессии Битрикс — это отдельная задача на бэкенде.

## Проверка

1. Сохраните приложение в Битрикс24, при необходимости нажмите **Установить** / откройте приложение из меню.
2. Должен загрузиться iframe с вашим доменом, редирект на раздел **Сценарии**.
3. В DevTools → Network проверьте, что запросы к `/api/` уходят на ваш сервер с нужными заголовками.

---

## Типичные проблемы

| Симптом | Что проверить |
|---------|----------------|
| Пустой iframe, ошибка в консоли про **frame** | CSP / `X-Frame-Options`, см. раздел Nginx выше. |
| **Mixed Content** | И SPA, и API только по **https**. |
| **502** на `/api/bitrix/install` | Доступность URL извне, POST, firewall. |
| Цикл **логин → снова логин** | Cookies / SameSite, что фронт и API на одном сайте или настроен общий домен для cookie. |
